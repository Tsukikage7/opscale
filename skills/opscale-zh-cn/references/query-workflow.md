# 查询工作流

多表指标、漏斗/留存、收入/退款计算或业务定义不清时读取本文件。

## 先看动态 Schema

使用 `opscale schema` 查看可用 schema、表和字段。对每个可能相关的事实表和关联表，先使用 `opscale describe <table>`，再写最终 SQL。

动态 schema 是事实来源。运营数据库会变化，不要依赖猜测的表名、ORM 约定或过期文档。

## 业务语义层

数据库元数据无法完整解释业务语义。回答指标时，要明确说明以下假设：

- 金额单位，例如分、元、美元或最小货币单位。
- 状态口径，例如 paid、completed、refunded。
- 时间字段，例如创建时间、支付时间、完成时间、退款时间。
- 软删除字段。
- 用户、订单、资源、支付、退款之间的 join 路径。

如果用户要求精确业务指标，而 schema 仍有歧义，先查看附近项目文档或代码；仍不清楚时，问一个简短问题。

## 指标规划

写 SQL 前先确定：

| 决策 | 示例 |
| --- | --- |
| 粒度 | 每天、每周、按账号、按套餐 |
| 人群/对象 | 全部用户、活跃用户、已支付订单、已完成资源 |
| 时间字段 | `created_at`、`paid_at`、`refunded_at`、`completed_at` |
| 状态过滤 | `paid`、`completed`、`active`，排除 `cancelled` |
| 金额单位 | 分、元、美元、最小货币单位 |
| 排除项 | 软删除记录、测试账号、内部用户 |

如果用户只说“最近”，且项目没有明确约定，默认使用最近 7 天，并说明假设。

## 执行模式

1. 运行 `opscale schema`。
2. 对候选表运行 `opscale describe <table>`。
3. 写 SQL，使用显式列名和清晰别名。
4. 多步骤指标优先使用 `WITH` CTE，不写难读的深层嵌套表达式。
5. 通过 `opscale run --sql` 执行。
6. 查询失败时，根据错误和 schema 输出修正。
7. 总结 JSON 结果。
8. 展示 SQL 和假设。

## 查询形态

- 先聚合，再列原始记录。
- 运营指标必须带明确时间范围。
- 时间序列和排行结果要有 `order by`。
- 结果规模应足够小，能放进回答。
- 避免 `select *`。
- 除非方言需要，避免不必要的数据库专属 SQL。

## 示例问题

```text
使用 opscale skill 查一下最近 7 天每天的付费订单数、GMV 和退款金额。
```

预期行为：

1. 确认粒度是按天，时间范围假设为最近 7 天。
2. 查看订单、支付、退款、用户、订阅相关表结构。
3. 验证状态、金额、时间字段。
4. 执行一个或多个只读聚合查询。
5. 解释结果、SQL 和业务假设。

## 安全基线

预期配置是只读数据库账号。CLI 使用 Node 数据库驱动，应用只读 SQL guard，并限制返回行数。

除非用户明确要求，否则不要绕过 CLI 使用 `psql`、临时数据库客户端或应用数据库凭据。
